## Contextual History — Tại Sao HTTP Tồn Tại?

### Câu chuyện: Từ CERN đến Internet hiện đại

**Năm 1989 — Tim Berners-Lee tại CERN:**

Tim Berners-Lee, một nhà khoa học tại CERN (Thụy Sĩ), muốn giải quyết một vấn đề đơn giản: các nhà khoa học cần **chia sẻ tài liệu nghiên cứu** giữa các máy tính. Ông phát minh ra 3 thứ:

1. **HTML** — ngôn ngữ để viết tài liệu có hyperlink
2. **URL** — địa chỉ duy nhất cho mỗi tài liệu
3. **HTTP** — protocol để truyền tài liệu giữa client và server

```
1989 ─── Tim Berners-Lee đề xuất "WorldWideWeb" tại CERN
  │       Mục đích: chia sẻ tài liệu khoa học!
  │
1991 ─── HTTP/0.9: CỰC KỲ ĐƠN GIẢN
  │       → Chỉ có GET
  │       → Chỉ trả HTML (không có header, không có status code!)
  │       → 1 request = 1 TCP connection → close ngay!
  │       Request:  "GET /page.html\r\n"
  │       Response: "<html>...</html>" (rồi close!)
  │
1996 ─── HTTP/1.0 (RFC 1945):
  │       → Thêm: Headers, Status codes, POST, HEAD
  │       → Thêm: Content-Type (không chỉ HTML nữa!)
  │       → VẪN: 1 request per connection (close sau mỗi response)
  │       → VẪN: mở close liên tục = CHẬM!
  │
1997 ─── HTTP/1.1 (RFC 2068, update RFC 2616, RFC 7230-7235):
  │       → Keep-Alive: MẶC ĐỊNH dùng connection liên tục!
  │       → Pipelining: gửi nhiều request KHÔNG ĐỢI response
  │       → Chunked transfer: streaming response
  │       → Host header: virtual hosting (1 IP → nhiều domain)
  │       → Cache control: ETag, If-Modified-Since
  │       → Content negotiation
  │       → ⚠️ VẪN: Head-of-line blocking!
  │       → Thống trị internet SUỐT 18 NĂM (1997-2015)!
  │
2009 ─── Google phát minh SPDY (đọc: "speedy")
  │       → Binary framing, multiplexing, header compression
  │       → Tiền thân của HTTP/2
  │
2015 ─── HTTP/2 (RFC 7540):
  │       → Binary protocol (thay vì text)
  │       → Multiplexing: nhiều requests trên 1 TCP connection!
  │       → HPACK: header compression
  │       → Server Push: server gửi resource TRƯỚC KHI client yêu cầu
  │       → ⚠️ NHƯNG: Head-of-line blocking ở TCP layer!
  │
2022 ─── HTTP/3 (RFC 9114):
         → QUIC thay TCP (chạy trên UDP)
         → 0-RTT connection setup
         → Không còn HOL blocking ở transport!
         → Connection migration (đổi WiFi → 4G giữ connection)
         → ⚠️ NHƯNG: CPU cost cao, middlebox compatibility issues
```

## 5 Whys: Tại Sao HTTP Được Thiết Kế Như Vậy?

```
╔═══════════════════════════════════════════════════════════════╗
║   5 WHYS — HTTP DESIGN DECISIONS                              ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  WHY 1: "Tại sao HTTP là stateless?"                         ║
║  → Vì server KHÔNG LƯU trạng thái giữa các requests.      ║
║    Mỗi request ĐỘC LẬP, chứa ĐẦY ĐỦ thông tin cần thiết.║
║                                                               ║
║  WHY 2: "Tại sao stateless? Lưu state không tiện hơn sao?" ║
║  → Vì web có HÀNG TRIỆU user đồng thời.                   ║
║    Lưu state cho mỗi user = memory KHỔNG LỒ!               ║
║    Server crash → MẤT TẤT CẢ state!                        ║
║    Stateless → BẤT KỲ server nào cũng handle được request!║
║                                                               ║
║  WHY 3: "Stateless thì làm sao login/session?"              ║
║  → Thêm state VÀO REQUEST: Cookies, JWT tokens!            ║
║    Client GỮ state → gửi kèm mỗi request.                 ║
║    Server chỉ cần VERIFY, không cần STORE!                  ║
║                                                               ║
║  WHY 4: "Nhưng session data vẫn cần lưu đâu đó?"          ║
║  → Lưu ở EXTERNAL store: Redis, Database.                   ║
║    Tách state khỏi server = HORIZONTAL SCALING!             ║
║    10 servers share 1 Redis = tất cả biết session!          ║
║                                                               ║
║  WHY 5: "Vậy stateless có nhược điểm gì?"                  ║
║  → Headers LẶP LẠI mỗi request! Cookies, Auth token,      ║
║    Accept headers... cùng data gửi hàng trăm lần/page!     ║
║    → HTTP/2 HPACK compression: nén headers!                 ║
║    → Nhưng vẫn có overhead vs WebSocket (stateful, 1 lần)  ║
║                                                               ║
║  ⭐ BÀI HỌC:                                                ║
║  Stateless = CỰC KỲ quan trọng cho scalability!            ║
║  → Load balancer: route request đến BẤT KỲ server nào!    ║
║  → Auto-scaling: thêm/bớt server tự do!                    ║
║  → Failure recovery: server crash → không mất state!       ║
║  → Đây là nguyên tắc CỐT LÕI của RESTful architecture!   ║
║                                                               ║
╚═══════════════════════════════════════════════════════════════╝
```

### 5 Whys: Tại Sao Cần HTTP Methods?

```
WHY 1: "Tại sao cần GET, POST, PUT, DELETE thay vì chỉ dùng POST?"
→ Vì SEMANTICS! Method nói cho server/proxy/cache BIẾT request
  này LÀM GÌ mà không cần đọc body.

WHY 2: "Tại sao semantics quan trọng?"
→ Vì CACHING + IDEMPOTENCY!
  GET = safe + idempotent → proxy CÓ THỂ cache response!
  POST = not safe, not idempotent → proxy KHÔNG cache!
  PUT/DELETE = idempotent → retry an toàn khi network lỗi!

WHY 3: "Idempotent nghĩa là gì, tại sao quan trọng?"
→ Idempotent = gọi 1 lần hay 100 lần KẾT QUẢ GIỐNG NHAU.
  Network timeout → retry → nếu idempotent: an toàn!
  GET /user/1: đọc 100 lần = kết quả giống nhau ✅
  DELETE /user/1: xóa 100 lần = user vẫn bị xóa ✅
  POST /orders: tạo 100 lần = 100 orders! ❌
  → POST cần idempotency key để tránh duplicate!

WHY 4: "Vậy tại sao nhiều API chỉ dùng POST cho mọi thứ?"
→ Vì developer LƯỜI hoặc không hiểu HTTP semantics!
  POST /getUser, POST /deleteUser, POST /createOrder...
  → Mất khả năng caching, retry, và proxy optimization!
  → GraphQL cũng dùng POST cho mọi thứ → trade-off!

WHY 5: "GraphQL dùng POST có sai không?"
→ KHÔNG SAI, nhưng là ĐÁNH ĐỔI có ý thức:
  → GraphQL đặt query logic TRONG body (POST body = query)
  → Cache phải ở application level (không dùng HTTP cache)
  → Trade-off: flexibility vs cacheability
  → Giải pháp: Persisted Queries + GET cho cacheable queries
```

### HTTP Methods — Semantics Quan Trọng

```
┌──────────┬───────┬────────────┬────────┬────────────────────────────────┐
│ Method   │ Safe? │ Idempotent?│ Body?  │ Ý nghĩa                       │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ GET      │ ✅    │ ✅         │ Không* │ Đọc resource                   │
│          │       │            │        │ → CACHEABLE bởi proxy/CDN!    │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ HEAD     │ ✅    │ ✅         │ Không  │ Giống GET nhưng KHÔNG có body │
│          │       │            │        │ → Check resource exists/size  │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ POST     │ ❌    │ ❌         │ Có     │ Tạo mới resource              │
│          │       │            │        │ → KHÔNG retry an toàn!        │
│          │       │            │        │ → Cần idempotency key!        │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ PUT      │ ❌    │ ✅         │ Có     │ Replace TOÀN BỘ resource      │
│          │       │            │        │ → Retry an toàn              │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ PATCH    │ ❌    │ ❌*        │ Có     │ Update MỘT PHẦN resource      │
│          │       │            │        │ *CÓ THỂ idempotent nếu       │
│          │       │            │        │ implementation đúng            │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ DELETE   │ ❌    │ ✅         │ Có thể │ Xóa resource                  │
│          │       │            │        │ → DELETE 2 lần = vẫn bị xóa │
├──────────┼───────┼────────────┼────────┼────────────────────────────────┤
│ OPTIONS  │ ✅    │ ✅         │ Không  │ Hỏi server hỗ trợ methods gì │
│          │       │            │        │ → CORS preflight request!     │
└──────────┴───────┴────────────┴────────┴────────────────────────────────┘

*GET với body: HTTP spec KHÔNG CẤM, nhưng hầu hết proxy/cache
IGNORE body của GET → ĐỪNG gửi body trong GET!
Elasticsearch dùng GET+body cho search → controversial!
```

### Status Codes — Phân Loại Chi Tiết

```
╔═══════════════════════════════════════════════════════════════╗
║   HTTP STATUS CODES — BACKEND ENGINEER PHẢI NHỚ             ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  1xx INFORMATIONAL (hiếm gặp):                               ║
║  ├── 100 Continue    → "Gửi body đi, header OK!"            ║
║  │   (dùng với Expect: 100-continue header)                  ║
║  ├── 101 Switching Protocols → WebSocket upgrade!            ║
║  └── 103 Early Hints → Preload resources (experimental)     ║
║                                                               ║
║  2xx SUCCESS:                                                 ║
║  ├── 200 OK          → Request thành công (GET, PUT, PATCH) ║
║  ├── 201 Created     → Resource đã tạo (POST)               ║
║  │   → SHOULD: Location header với URL của resource mới!    ║
║  ├── 202 Accepted    → Request nhận, xử lý SAU (async!)     ║
║  │   → Dùng cho job queue: POST /export → 202 + job ID     ║
║  ├── 204 No Content  → Thành công, không có body (DELETE)   ║
║  └── 206 Partial     → Range request (video streaming!)     ║
║                                                               ║
║  3xx REDIRECTION:                                             ║
║  ├── 301 Moved Permanently  → URL mới VĨNH VIỄN            ║
║  │   → Browser/proxy CACHE redirect này! SEO: juice chuyển!║
║  ├── 302 Found (Temporary)  → URL mới TẠM THỜI            ║
║  │   → KHÔNG cache! Lần sau vẫn request URL cũ!           ║
║  ├── 304 Not Modified → "Resource CHƯA THAY ĐỔI, dùng    ║
║  │   cache đi!" (với ETag/If-Modified-Since)                ║
║  │   → TIẾT KIỆM bandwidth! Không gửi body!               ║
║  └── 307/308 → Giống 302/301 nhưng GIỮ NGUYÊN method!     ║
║      → 302 có thể đổi POST→GET! 307 giữ POST!            ║
║                                                               ║
║  4xx CLIENT ERROR (lỗi của CLIENT):                          ║
║  ├── 400 Bad Request     → Request SAI format/validation    ║
║  ├── 401 Unauthorized    → CHƯA XÁC THỰC (chưa login!)    ║
║  │   → Tên gây hiểu lầm! Đúng ra phải là "Unauthenticated"║
║  ├── 403 Forbidden       → ĐÃ XÁC THỰC nhưng KHÔNG CÓ    ║
║  │   QUYỀN! (logged in, but not admin)                      ║
║  ├── 404 Not Found       → Resource không tồn tại          ║
║  ├── 405 Method Not Allow→ Endpoint tồn tại nhưng method   ║
║  │   không hỗ trợ (POST lên GET-only endpoint)             ║

║  ├── 422 Unprocessable   → Syntax đúng nhưng logic SAI    ║
║  │   → Validation error: "email invalid", "age < 0"        ║
║  ├── 429 Too Many Requests → Rate limiting!                  ║
║  │   → Retry-After header: bao lâu client nên đợi!        ║

║  │   → ĐỪNG expose stack trace cho client! (security!)      ║

║  │   → Retry-After header! Client nên back off!            ║

║  → 4xx = CLIENT sai → log WARNING, không alert!            ║
║  → 5xx = SERVER sai → log ERROR, ALERT ngay!               ║
```

```go
// Go: HTTP server minh họa status codes + best practices
package main

import (
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "strconv"
    "strings"
    "sync"
    "time"
)

// Response envelope chuẩn
type APIResponse struct {
    Data  interface{} `json:"data,omitempty"`
    Error *APIError   `json:"error,omitempty"`
    Meta  *Meta       `json:"meta,omitempty"`
}

type APIError struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}

type Meta struct {
    RequestID string `json:"request_id"`
    Took      string `json:"took"`
}

type User struct {
    ID    int    `json:"id"`
    Name  string `json:"name"`
    Email string `json:"email"`
}

var (
    users = map[int]*User{
        1: {ID: 1, Name: "Alice", Email: "alice@example.com"},
        2: {ID: 2, Name: "Bob", Email: "bob@example.com"},
    }
    mu     sync.RWMutex
    nextID = 3
)

func main() {
    mux := http.NewServeMux()

    // RESTful routes
    mux.HandleFunc("/api/users", handleUsers)
    mux.HandleFunc("/api/users/", handleUserByID)

    server := &http.Server{
        Addr:         ":8080",
        Handler:      loggingMiddleware(mux),
        ReadTimeout:  5 * time.Second,
        WriteTimeout: 10 * time.Second,
        IdleTimeout:  120 * time.Second,
    }

    log.Println("Server listening on :8080")
    log.Fatal(server.ListenAndServe())
}

func handleUsers(w http.ResponseWriter, r *http.Request) {
    switch r.Method {
    case http.MethodGet:
        // GET /api/users → 200 + list
        mu.RLock()
        list := make([]*User, 0, len(users))
        for _, u := range users {
            list = append(list, u)
        }
        mu.RUnlock()
        writeJSON(w, http.StatusOK, list)

    case http.MethodPost:
        // POST /api/users → 201 Created + Location header
        var u User
        if err := json.NewDecoder(r.Body).Decode(&u); err != nil {
            writeError(w, http.StatusBadRequest, "INVALID_JSON",
                "Request body is not valid JSON")
            return
        }
        if u.Email == "" {
            writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR",
                "Email is required") // 422, không phải 400!
            return
        }
        mu.Lock()
        u.ID = nextID
        nextID++
        users[u.ID] = &u
        mu.Unlock()

        // 201 Created + Location header → đúng REST semantics!
        w.Header().Set("Location", fmt.Sprintf("/api/users/%d", u.ID))
        writeJSON(w, http.StatusCreated, u)

    default:
        // 405 Method Not Allowed
        w.Header().Set("Allow", "GET, POST") // NÊN có Allow header!
        writeError(w, http.StatusMethodNotAllowed, "METHOD_NOT_ALLOWED",
            fmt.Sprintf("Method %s not allowed", r.Method))
    }
}

func handleUserByID(w http.ResponseWriter, r *http.Request) {
    // Parse ID from URL
    idStr := strings.TrimPrefix(r.URL.Path, "/api/users/")
    id, err := strconv.Atoi(idStr)
    if err != nil {
        writeError(w, http.StatusBadRequest, "INVALID_ID", "ID must be integer")
        return
    }

    switch r.Method {
    case http.MethodGet:
        mu.RLock()
        u, exists := users[id]
        mu.RUnlock()
        if !exists {
            writeError(w, http.StatusNotFound, "NOT_FOUND",
                fmt.Sprintf("User %d not found", id))
            return
        }
        // Cache control: ETag cho conditional GET
        etag := fmt.Sprintf(`"%d-%s"`, u.ID, u.Email)
        w.Header().Set("ETag", etag)
        // Check If-None-Match → 304 Not Modified!
        if r.Header.Get("If-None-Match") == etag {
            w.WriteHeader(http.StatusNotModified) // 304: không gửi body!
            return
        }
        writeJSON(w, http.StatusOK, u)

    case http.MethodDelete:
        mu.Lock()
        if _, exists := users[id]; !exists {
            mu.Unlock()
            writeError(w, http.StatusNotFound, "NOT_FOUND",
                fmt.Sprintf("User %d not found", id))
            return
        }
        delete(users, id)
        mu.Unlock()
        w.WriteHeader(http.StatusNoContent) // 204: no body for DELETE

    default:
        w.Header().Set("Allow", "GET, DELETE")
        writeError(w, http.StatusMethodNotAllowed, "METHOD_NOT_ALLOWED",
            fmt.Sprintf("Method %s not allowed", r.Method))
    }
}

func writeJSON(w http.ResponseWriter, status int, data interface{}) {
    w.Header().Set("Content-Type", "application/json; charset=utf-8")
    w.WriteHeader(status)
    json.NewEncoder(w).Encode(APIResponse{Data: data})
}

func writeError(w http.ResponseWriter, status int, code, msg string) {
    w.Header().Set("Content-Type", "application/json; charset=utf-8")
    w.WriteHeader(status)
    json.NewEncoder(w).Encode(APIResponse{
        Error: &APIError{Code: code, Message: msg},
    })
}

func loggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        // Request ID cho distributed tracing
        requestID := r.Header.Get("X-Request-ID")
        if requestID == "" {
            requestID = fmt.Sprintf("%d", time.Now().UnixNano())
        }
        w.Header().Set("X-Request-ID", requestID)

        next.ServeHTTP(w, r)

        log.Printf("[%s] %s %s %v", requestID, r.Method, r.URL.Path,
            time.Since(start))
    })
}
```

## §4. HTTP/1.1 — Connection Management & Vấn Đề

### Keep-Alive — Tại Sao Quan Trọng?


## §5. HTTP/2 — Binary Revolution

### Tại sao HTTP/2 ra đời? (5 Whys)

```
WHY 1: "Tại sao cần HTTP/2?"
→ Vì HTTP/1.1 có Head-of-line Blocking: 1 slow response
  block TẤT CẢ requests trên cùng connection!

WHY 2: "Tại sao HOL blocking là vấn đề lớn?"
→ Vì modern web pages cần 80-100+ resources!
  6 connections × HOL blocking = vẫn CHẬM!

WHY 3: "Sao không mở thêm connections?"
→ Mỗi TCP connection = memory + 3-way handshake + slow start.
  100 connections = 100× overhead! Server KHÔNG CHỊU NỔI!

WHY 4: "Pipelining (gửi nhiều request không đợi) thì sao?"
→ Pipelining broken bởi design:
  → Response PHẢI theo đúng THỨ TỰ request!
  → 1 slow response block TOÀN BỘ pipeline!
  → Proxy/server implementation BAD → Chrome TẮT pipelining!

WHY 5: "Vậy HTTP/2 fix thế nào?"
→ MULTIPLEXING: chia data thành FRAMES nhỏ,
  interleave FRAMES của nhiều streams trên 1 connection!
  → Không còn HOL blocking ở HTTP layer!
  → 1 connection cho TẤT CẢ requests!
```

### HPACK — Header Compression

```
╔═══════════════════════════════════════════════════════════════╗
║   HPACK — TẠI SAO CẦN NÉN HEADERS?                         ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  HTTP/1.1: Headers gửi LẠP ĐI LẶP LẠI mỗi request!       ║
║                                                               ║
║  Request 1:                                                   ║
║  Host: api.example.com                    (20 bytes)          ║
║  User-Agent: Mozilla/5.0 ... (dài!)      (200 bytes)         ║
║  Accept: application/json                 (24 bytes)          ║
║  Authorization: Bearer eyJhbGciOi...      (500 bytes)         ║
║  Cookie: session=very-long-string         (300 bytes)         ║
║  Total headers: ~1,044 bytes                                  ║
║                                                               ║
║  Request 2-100: CÙNG HEADERS! 1,044 × 100 = 104,400 bytes! ║
║  → Hơn 100 KB chỉ cho HEADERS!                              ║
║                                                               ║
║  HPACK giải quyết:                                           ║
║  ────────────────                                            ║
║  1. Static Table: 61 common headers đã được index            ║
║     ":method: GET" = index 2 = 1 byte!                       ║
║     ":status: 200" = index 8 = 1 byte!                       ║
║                                                               ║
║  2. Dynamic Table: headers THỰC TẾ được ghi nhớ             ║
║     Request 1: "Authorization: Bearer eyJ..." = entry #62   ║
║     Request 2: gửi index 62 = 1-2 bytes!                    ║
║     → Từ 500 bytes → 2 bytes! Nén 99.6%!                   ║
║                                                               ║
║  3. Huffman Encoding: nén value strings                      ║
║     "application/json" = 16 bytes → 12 bytes                ║
║                                                               ║
║  Kết quả:                                                     ║
║  → Request 2-100: headers chỉ ~10-20 bytes mỗi request!    ║
║  → Tiết kiệm ~95% bandwidth cho headers!                    ║
║  → Đặc biệt quan trọng cho mobile (bandwidth đắt!)         ║
║                                                               ║
║  ⚠️ Security concern: CRIME/BREACH attack                   ║
║  → Compression + secret in same context = info leak!        ║
║  → HPACK designed to be safe (no cross-context compression) ║
```

```
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
```

```
║  NHƯNG tất cả streams đi qua 1 TCP connection!             ║

║  → Packet 3 mất → TCP buffer packet 4,5,6 → ĐỢI packet 3!║

║  → HTTP/2 dùng 1 connection → loss block TOÀN BỘ!          ║
```

## §7. Cookies, Sessions & Authentication

### Tại sao cần Cookies? (5 Whys)

```
WHY 1: "Tại sao cần cookies?"
→ Vì HTTP là STATELESS! Server không nhớ client giữa requests!

WHY 2: "Tại sao server cần nhớ?"
→ Vì cần: login, shopping cart, preferences, tracking...
  "Tôi là ai? Giỏ hàng tôi có gì? Dark mode hay light?"

WHY 3: "Sao không gửi user info mỗi request?"
→ ĐÓ CHÍNH LÀ cookie! Server set cookie → browser gửi lại tự động!
  Set-Cookie: session=abc123 → Browser gửi Cookie: session=abc123
  MỖI request đến cùng domain!

WHY 4: "Cookie vs JWT — cái nào tốt hơn?"
→ Không có "tốt hơn", chỉ có TRADE-OFF:

Cookie + Server Session:
  ✅ Server kiểm soát hoàn toàn (revoke ngay!)
  ✅ HttpOnly: JS không đọc được (chống XSS)
  ❌ Cần session store (Redis)
  ❌ Sticky session hoặc shared store → phức tạp
  ❌ CSRF vulnerable (cookie gửi tự động!)

JWT (stateless token):
  ✅ Không cần server-side storage
  ✅ Tự chứa info (claims): user ID, role, expiry
  ✅ Microservices: bất kỳ service nào đều verify được!
  ❌ KHÔNG THỂ revoke (cho đến khi hết hạn!)
  ❌ Token size lớn (header + payload + signature)
  ❌ Nếu lộ secret key → toàn bộ hệ thống bị compromise!

WHY 5: "Vậy production dùng gì?"
→ HYBRID: JWT (short-lived, 15 phút) + Refresh Token (longer)!
  → Access token hết hạn nhanh → giảm risk nếu bị lộ!
  → Refresh token trong HttpOnly cookie → an toàn hơn!
  → Rotate refresh token mỗi lần dùng → detect stolen token!
```

```go
import (
    "fmt"
    "log"
    "net/http"
    "sync"

    "github.com/gorilla/websocket"
)

var upgrader = websocket.Upgrader{
    ReadBufferSize:  1024,
    WriteBufferSize: 1024,
    CheckOrigin: func(r *http.Request) bool {
        // Production: validate origin!
        return true
    },

// Connection manager — broadcast to all clients
type Hub struct {
    clients    map[*websocket.Conn]bool
    broadcast  chan []byte
    register   chan *websocket.Conn
    unregister chan *websocket.Conn
    mu         sync.RWMutex
}

func newHub() *Hub {
    return &Hub{
        clients:    make(map[*websocket.Conn]bool),
        broadcast:  make(chan []byte, 256),
        register:   make(chan *websocket.Conn),
        unregister: make(chan *websocket.Conn),
    }
}

func (h *Hub) run() {
    for {
        select {
        case conn := <-h.register:
            h.mu.Lock()
            h.clients[conn] = true
            h.mu.Unlock()
            log.Printf("Client connected: %s (total: %d)",
                conn.RemoteAddr(), len(h.clients))

        case conn := <-h.unregister:
            h.mu.Lock()
            if _, ok := h.clients[conn]; ok {
                delete(h.clients, conn)
                conn.Close()
            }
            h.mu.Unlock()

        case msg := <-h.broadcast:
            h.mu.RLock()
            for conn := range h.clients {
                if err := conn.WriteMessage(websocket.TextMessage, msg); err != nil {
                    log.Printf("Write error to %s: %v", conn.RemoteAddr(), err)
                    conn.Close()
                    delete(h.clients, conn)
                }
            }
            h.mu.RUnlock()
        }
    }
}

func main() {
    hub := newHub()
    go hub.run()

    http.HandleFunc("/ws", func(w http.ResponseWriter, r *http.Request) {
        // HTTP → WebSocket upgrade (101 Switching Protocols)
        conn, err := upgrader.Upgrade(w, r, nil)
        if err != nil {
            log.Println("Upgrade error:", err)
            return
        }
        hub.register <- conn

        // Read loop (mỗi connection = 1 goroutine)
        defer func() { hub.unregister <- conn }()
        for {
            _, msg, err := conn.ReadMessage()
            if err != nil {
                break
            }
            // Broadcast to all clients
            hub.broadcast <- []byte(fmt.Sprintf("[%s]: %s",
                conn.RemoteAddr(), msg))
        }
    })

    log.Println("WebSocket server on :8080/ws")
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

## §10. Reverse Engineering — HTTP Từ Zero

### Go: Build HTTP Server Từ TCP Socket

```go
// Reverse Engineering: HTTP server TỪ TCP socket
// "What I cannot create, I do not understand" — Feynman
// Mục đích: HIỂU HTTP protocol, KHÔNG dùng cho production!
package main

import (
    "bufio"
    "fmt"
    "net"
    "strings"
    "time"
)

func main() {
    listener, err := net.Listen("tcp", ":9090")
    if err != nil {
        panic(err)
    }
    fmt.Println("[HTTP Server from scratch] Listening on :9090")

    for {
        conn, err := listener.Accept()
        if err != nil {
            continue
        }
        go handleHTTPConnection(conn)
    }
}

func handleHTTPConnection(conn net.Conn) {
    defer conn.Close()
    conn.SetDeadline(time.Now().Add(5 * time.Second))

    reader := bufio.NewReader(conn)

    // 1. Parse REQUEST LINE: "GET /path HTTP/1.1\r\n"
    requestLine, err := reader.ReadString('\n')
    if err != nil {
        return
    }
    requestLine = strings.TrimSpace(requestLine)
    parts := strings.SplitN(requestLine, " ", 3)
    if len(parts) != 3 {
        sendResponse(conn, 400, "Bad Request", "Invalid request line")
        return
    }
    method, path, version := parts[0], parts[1], parts[2]
    fmt.Printf("[REQUEST] %s %s %s\n", method, path, version)

    // 2. Parse HEADERS (đọc cho đến dòng trống)
    headers := map[string]string{}
    for {
        line, err := reader.ReadString('\n')
        if err != nil {
            return
        }
        line = strings.TrimSpace(line)
        if line == "" {
            break // Dòng trống = hết headers!
        }
        colonIdx := strings.IndexByte(line, ':')
        if colonIdx > 0 {
            key := strings.TrimSpace(line[:colonIdx])
            value := strings.TrimSpace(line[colonIdx+1:])
            headers[strings.ToLower(key)] = value
        }
    }

    // 3. Route request (mini router)
    switch {
    case method == "GET" && path == "/":
        body := `{"message": "Hello from scratch HTTP server!", "time": "` +
            time.Now().Format(time.RFC3339) + `"}`
        sendJSON(conn, 200, "OK", body)

    case method == "GET" && path == "/health":
        sendJSON(conn, 200, "OK", `{"status": "healthy"}`)

    default:
        sendResponse(conn, 404, "Not Found",
            fmt.Sprintf("Path %s not found", path))
    }
}

func sendResponse(conn net.Conn, status int, statusText, body string) {
    // HTTP Response = Status Line + Headers + CRLF + Body
    response := fmt.Sprintf("HTTP/1.1 %d %s\r\n"+
        "Content-Type: text/plain; charset=utf-8\r\n"+
        "Content-Length: %d\r\n"+
        "Connection: close\r\n"+
        "Server: GoFromScratch/1.0\r\n"+
        "\r\n"+ // ← DÒNG TRỐNG = hết headers!
        "%s",
        status, statusText, len(body), body)

    conn.Write([]byte(response))
}

func sendJSON(conn net.Conn, status int, statusText, body string) {
    response := fmt.Sprintf("HTTP/1.1 %d %s\r\n"+
        "Content-Type: application/json; charset=utf-8\r\n"+
        "Content-Length: %d\r\n"+
        "Connection: close\r\n"+
        "X-Powered-By: GoFromScratch\r\n"+
        "\r\n"+
        "%s",
        status, statusText, len(body), body)

    conn.Write([]byte(response))
}

// Chạy: go run main.go
// Test: curl -v http://localhost:9090/
//       curl -v http://localhost:9090/health
//       curl -v http://localhost:9090/notfound
//
// BÀI HỌC khi implement từ zero:
// 1. HTTP thực chất là TEXT protocol trên TCP
// 2. \r\n là separator bắt buộc
// 3. Dòng trống phân cách headers và body
// 4. Content-Length BẮT BUỘC (trừ chunked)
// 5. net/http của Go xử lý TẤT CẢ edge cases này cho bạn!
```

## Tổng Kết — HTTP Cheat Sheet

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   HTTP CHEAT SHEET — BACKEND ENGINEER ESSENTIALS                         ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  CORE:                                                                    ║
║  → Stateless, request-response, text (1.1) / binary (2, 3)              ║
║  → Methods: GET (safe+idempotent), POST (not), PUT/DELETE (idempotent)  ║
║  → Status: 2xx=OK, 3xx=redirect, 4xx=client error, 5xx=server error    ║
║                                                                           ║
║  HTTP/1.1 (1997):                                                        ║
║  → Keep-alive default, Host header, chunked transfer                    ║
║  → ⚠️ HOL blocking, 6 connections/domain limit                         ║
║                                                                           ║
║  HTTP/2 (2015):                                                          ║
║  → Binary framing, multiplexing, HPACK header compression              ║
║  → 1 TCP connection, streams independent at HTTP layer                  ║
║  → ⚠️ TCP HOL blocking still exists!                                   ║
║                                                                           ║
║  HTTP/3 (2022):                                                          ║
║  → QUIC over UDP, 0-RTT, connection migration                          ║
║  → Independent stream reliability, built-in TLS 1.3                     ║
║  → ⚠️ CPU cost, middlebox compat, debugging harder                     ║
║                                                                           ║
║  CACHING:                                                                 ║
║  → Cache-Control: public/private/no-cache/no-store/max-age              ║
║  → ETag + If-None-Match → 304 Not Modified (save bandwidth!)           ║
║  → Static: immutable + hash filename. API: no-cache + ETag!            ║
║                                                                           ║
║  SECURITY HEADERS:                                                        ║
║  → HSTS, CSP, X-Content-Type-Options, X-Frame-Options                  ║
║  → Cookie: HttpOnly + Secure + SameSite=Lax                            ║
║  → CORS: specify domain, not * with credentials!                        ║
║                                                                           ║
║  AUTHENTICATION:                                                          ║
║  → JWT (short-lived) + Refresh Token (HttpOnly cookie)                  ║
║  → 401=not authenticated, 403=not authorized                            ║
║                                                                           ║
║  WEBSOCKET vs SSE vs HTTP:                                               ║
║  → WebSocket: bidirectional, real-time (chat, gaming)                   ║
║  → SSE: server→client only, auto-reconnect (notifications)             ║
║  → HTTP: request-response (REST APIs, file upload)                      ║
║                                                                           ║
║  GO PRODUCTION:                                                           ║
║  → ALWAYS set ReadTimeout, WriteTimeout, IdleTimeout!                   ║
║  → HTTP/2 auto-enabled with TLS (net/http)                              ║
║  → MaxIdleConnsPerHost default=2 → INCREASE for high traffic!          ║
║  → ALWAYS: io.Copy(io.Discard, resp.Body); resp.Body.Close()           ║
║                                                                           ║
║  INTERVIEW QUESTIONS:                                                     ║
║  → "HTTP stateless là gì? Scale thế nào với session?"                  ║
```
