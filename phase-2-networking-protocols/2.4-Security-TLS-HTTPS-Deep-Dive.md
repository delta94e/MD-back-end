## Contextual History — Tại Sao HTTPS Tồn Tại?

### Câu chuyện: Từ HTTP Plaintext đến TLS 1.3

**Những năm 1990 — Internet "trần trụi":**

HTTP/1.0 gửi MỌI THỨ dưới dạng plaintext. Ai ở giữa (ISP, WiFi
cùng quán cafe, router bị hack) NHÌN THẤY TẤT CẢ: mật khẩu, thẻ
tín dụng, email, cookie. Internet ban đầu là mạng NGHIÊN CỨU, tin tưởng
lẫn nhau — không ai nghĩ đến bảo mật.

```
1994 ─── Netscape tạo SSL 1.0 (KHÔNG bao giờ public — quá nhiều lỗi!)
  │
1995 ─── SSL 2.0 released (NHIỀU lỗ hổng → deprecated!)
  │       → Man-in-the-middle, cipher downgrade attacks
  │
1996 ─── SSL 3.0 (thiết kế lại hoàn toàn bởi Paul Kocher)
  │       → Tốt hơn nhiều, nhưng vẫn có POODLE attack sau này
  │
1999 ─── TLS 1.0 (RFC 2246) — SSL đổi tên thành TLS!
  │       → "SSL" chỉ là tên CŨ, thực tế KHÔNG AI dùng SSL nữa!
  │       → Nhưng mọi người vẫn gọi "SSL certificate" → gây hiểu lầm!
  │
2006 ─── TLS 1.1 — fix BEAST attack
  │
2008 ─── TLS 1.2 (RFC 5246) — hiện TẠI vẫn được dùng rộng rãi
  │       → AEAD ciphers (AES-GCM)
  │       → SHA-256 (thay SHA-1)
  │       → Nhưng: handshake 2 RTT = CHẬM!
  │
2013 ─── Edward Snowden leak NSA mass surveillance
  │       → Internet chấn động! Mọi người nhận ra:
  │         "Nếu KHÔNG mã hóa, CHÍNH PHỦ đọc ĐƯỢC!"
  │       → Google: HTTPS as ranking signal!
  │       → Let's Encrypt project bắt đầu!
  │
2014 ─── Heartbleed (CVE-2014-0160) — lỗi nghiêm trọng nhất lịch sử!
  │       → Bug trong OpenSSL: đọc 64KB memory server!
  │       → Passwords, private keys, session data LEAKED!
  │       → 17% HTTPS servers bị ảnh hưởng!
  │       → BÀI HỌC: 1 bug trong crypto library = disaster toàn cầu!
  │
2015 ─── Let's Encrypt launched — certificate MIỄN PHÍ!
  │       → Trước: SSL cert = $100-1000/năm → chỉ site lớn dùng
  │       → Sau: MIỄN PHÍ + auto-renew → HTTPS cho MỌI NGƯỜI!
  │       → HTTPS adoption: 30% (2015) → 95%+ (2025)
  │
2018 ─── TLS 1.3 (RFC 8446) — CÁCH MẠNG!
  │       → 1-RTT handshake (thay vì 2-RTT)
  │       → 0-RTT resumption (data trong packet đầu tiên!)
  │       → Loại bỏ TOÀN BỘ insecure ciphers (RSA key exchange, CBC, RC4)
  │       → Chỉ giữ: ECDHE + AEAD + SHA-256+
  │       → Encrypted handshake (metadata cũng mã hóa!)
  │
2020 ─── TLS 1.0 & 1.1 officially DEPRECATED
  │       → Browsers không hỗ trợ nữa!
  │
2025 ─── Hiện tại:
         → TLS 1.2 + 1.3 chiếm >99% HTTPS traffic
         → HTTP (không S) bị Chrome đánh dấu "Not Secure"
         → Post-Quantum Cryptography chuẩn bị cho quantum computing!
```

```
╔═══════════════════════════════════════════════════════════════╗
║   5 WHYS — TẠI SAO HTTP KHÔNG ĐỦ?                           ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  WHY 1: "Tại sao cần HTTPS?"                                ║
║  → Vì HTTP gửi plaintext! Bất kỳ ai ở giữa (ISP, WiFi,   ║
║    hacker) đọc ĐƯỢC password, credit card, session cookie!  ║
║                                                               ║
║  WHY 2: "Ai ở giữa? Tại sao họ đọc được?"                 ║
║  → Man-in-the-Middle (MITM)! Network traffic đi qua:       ║
║    WiFi router → ISP → backbone → ISP → server             ║
║    BẤT KỲ HOP NÀO có thể sniff/modify data!               ║
║    → ISP inject quảng cáo vào HTTP pages!                   ║
║    → Hacker tại WiFi quán cafe capture cookies!              ║
║    → Thậm chí CHÍNH PHỦ surveillance (Snowden)!             ║
║                                                               ║
║  WHY 3: "Mã hóa thì an toàn? Sao biết đang nói với server ║
║          THẬT không phải hacker giả?"                        ║
║  → CERTIFICATES! Server chứng minh danh tính bằng:         ║
║    Certificate ký bởi CA (Certificate Authority) đáng tin   ║
║    → Browser có sẵn danh sách CA tin cậy (~150 CAs)        ║
║    → CA kiểm tra bạn SỞ HỮU domain trước khi ký!          ║
║                                                               ║
║  WHY 4: "Nếu CA bị hack thì sao? Tin CA vô điều kiện?"    ║
║  → ĐÃ XẢY RA! DigiNotar CA bị hack 2011 → phát hành      ║
║    certificate giả cho *.google.com!                         ║
║    → Giải pháp: Certificate Transparency (CT logs)          ║
║    → Certificate Pinning (app hardcode expected cert)        ║
║    → CAA DNS record (chỉ CA nào được phép ký cho domain)   ║
║    → Certificate revocation: CRL, OCSP                       ║
║                                                               ║
║  WHY 5: "Vậy HTTPS giải quyết MỌI THỨ?"                   ║
║  → KHÔNG! HTTPS chỉ bảo vệ TRANSPORT (data in transit):   ║
║    ✅ Mã hóa (không đọc được)                               ║
║    ✅ Integrity (không thay đổi được)                        ║
║    ✅ Authentication (đúng server)                            ║
║    ❌ KHÔNG bảo vệ data at rest (cần encrypt DB!)           ║
║    ❌ KHÔNG bảo vệ khỏi XSS/CSRF (web vulnerabilities!)   ║
║    ❌ KHÔNG bảo vệ khỏi SQL injection!                      ║
║    ❌ KHÔNG bảo vệ against weak passwords!                  ║
║    → Security = NHIỀU TẦNG, HTTPS chỉ là 1 TẦNG!          ║
║                                                               ║
```

### Symmetric vs Asymmetric Encryption

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   SYMMETRIC vs ASYMMETRIC — TẠI SAO CẦN CẢ HAI?                        ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  SYMMETRIC (Khóa đối xứng):                                            ║
║  ────────────────────────                                                ║
║  CÀI GÌ: Cùng 1 key để encrypt VÀ decrypt                             ║
║  GIỐNG NHƯ: 1 ổ khóa, 2 bản chìa khóa giống nhau                     ║
║                                                                           ║
║  Plaintext ──►[AES-256-GCM]──► Ciphertext                               ║
║                    ▲                  │                                    ║
║                    │                  ▼                                    ║
║                 Key: abc123    [AES-256-GCM] ──► Plaintext               ║
║                                    ▲                                      ║
║                                    │                                      ║
║                                 Key: abc123 (CÙNG KEY!)                  ║
║                                                                           ║
║  ✅ NHANH! AES-256: ~1 GB/s trên hardware                              ║
║  ✅ Overhead thấp                                                        ║
║  ❌ VẤN ĐỀ: Làm sao gửi key AN TOÀN qua internet?                    ║
║     → Nếu gửi key qua plaintext → hacker lấy key → xong!             ║
║     → "Key distribution problem" — BÀI TOÁN CỐT LÕI!                 ║
║                                                                           ║
║  Algorithms:                                                              ║
║  ┌────────────────────┬──────────┬──────────────────────────┐           ║
║  │ Algorithm          │ Key size │ Status                    │           ║
║  ├────────────────────┼──────────┼──────────────────────────┤           ║
║  │ AES-128-GCM        │ 128 bit  │ ✅ Secure, fast (HW accel)│          ║
║  │ AES-256-GCM        │ 256 bit  │ ✅ Government grade       │          ║
║  │ ChaCha20-Poly1305  │ 256 bit  │ ✅ Mobile-friendly (no HW)│          ║
║  │ RC4                │ various  │ ❌ BROKEN! Đừng dùng!    │          ║
║  │ DES / 3DES         │ 56/168   │ ❌ BROKEN! Đừng dùng!    │          ║
║  └────────────────────┴──────────┴──────────────────────────┘           ║
║                                                                           ║
║  ASYMMETRIC (Khóa bất đối xứng):                                       ║
║  ──────────────────────────────                                          ║
║  CÁI GÌ: 2 keys khác nhau — Public Key + Private Key                  ║
║  GIỐNG NHƯ: Hộp thư công cộng — ai cũng BỎ THƯ VÀO được,            ║
║             chỉ CHỦ có key MỞ RA đọc!                                  ║
║                                                                           ║
║  Encrypt:     Plaintext ──[Public Key]──► Ciphertext                    ║
║  Decrypt:     Ciphertext ──[Private Key]──► Plaintext                   ║
║  Sign:        Data ──[Private Key]──► Signature                         ║
║  Verify:      Signature ──[Public Key]──► Valid/Invalid                 ║
║                                                                           ║
║  ✅ Giải quyết key distribution! Public key = công khai!               ║
║  ✅ Digital signatures: chứng minh AI ký data                          ║
║  ❌ CHẬM! RSA-2048: ~1000× chậm hơn AES!                               ║
║  ❌ Không dùng cho bulk data encryption!                                 ║
║                                                                           ║
║  → GIẢI PHÁP TLS: KẾT HỢP CẢ HAI!                                    ║
║  1. Asymmetric (ECDHE) để trao đổi KEY an toàn                        ║
║  2. Symmetric (AES) để mã hóa DATA nhanh                              ║
║  → "Hybrid encryption" — best of both worlds!                          ║
║                                                                           ║

```

### Diffie-Hellman Key Exchange — Phép Màu Toán Học

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   DIFFIE-HELLMAN KEY EXCHANGE — HIỂU QUA PHÉP TƯ DUY                   ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  BÀI TOÁN: Alice và Bob muốn tạo CHUNG 1 secret key,                  ║
║  NHƯNG hacker Eve đang nghe TẤT CẢ communication!                      ║
║                                                                           ║
║  PHÉP SO SÁNH MÀU SƠN:                                                  ║
║                                                                           ║
║  1. Alice và Bob thống nhất: Màu CHUNG = VÀNG (public!)                ║
║     → Eve cũng biết màu vàng. OK, không sao!                           ║
║                                                                           ║
║  2. Alice chọn MÀU BÍ MẬT = ĐỎ (chỉ Alice biết!)                     ║
║     Bob chọn MÀU BÍ MẬT = XANH (chỉ Bob biết!)                        ║
║                                                                           ║
║  3. Alice trộn: VÀNG + ĐỎ = CAM → gửi cho Bob                        ║
║     Bob trộn: VÀNG + XANH = XANH LÁ → gửi cho Alice                  ║
║     → Eve thấy CAM và XANH LÁ. Nhưng KHÔNG THỂ tách                  ║
║       màu ra! (Đây là "discrete logarithm problem")                    ║
║                                                                           ║
║  4. Alice nhận XANH LÁ + trộn thêm ĐỎ = MÀU NÂU SECRET!             ║
║     Bob nhận CAM + trộn thêm XANH = MÀU NÂU SECRET!                   ║
║     → CẢ HAI có CÙNG MÀU NÂU mà Eve KHÔNG THỂ tạo ra!               ║
║                                                                           ║
║  TOÁN HỌC (đơn giản hóa):                                              ║
║  ───────────────────────                                                 ║
║  Public: g = 5, p = 23 (prime)                                          ║
║  Alice secret: a = 6 → A = g^a mod p = 5^6 mod 23 = 8                  ║
║  Bob secret:   b = 15 → B = g^b mod p = 5^15 mod 23 = 19              ║
║                                                                           ║
║  Alice gửi A=8, Bob gửi B=19 (public, Eve thấy!)                       ║
║                                                                           ║
║  Shared secret:                                                           ║
║  Alice: B^a mod p = 19^6 mod 23 = 2                                     ║
║  Bob:   A^b mod p = 8^15 mod 23 = 2                                     ║
║  → CÙNG KẾT QUẢ: 2! Đây là shared secret key!                        ║
║                                                                           ║
║  Eve biết: g=5, p=23, A=8, B=19                                         ║
║  Eve cần tìm: a hoặc b (discrete log problem = CỰC KHÓ!)              ║
║  → Với số THẬT (256-bit numbers), computers hiện tại                    ║
║    cần HÀNG TỶ NĂM để brute force!                                     ║
║                                                                           ║
║  ECDHE (Elliptic Curve DHE):                                             ║
║  → Cùng nguyên lý nhưng dùng Elliptic Curves                           ║
║  → 256-bit ECDHE = bảo mật tương đương RSA 3072-bit                   ║
║  → NHANH hơn RSA, key NGẮN hơn → tốt cho mobile/IoT!                 ║
║  → "E" = Ephemeral: key MỚI mỗi session → Forward Secrecy!           ║
║                                                                           ║
╚═══════════════════════════════════════════════════════════════════════════╝
```

### Forward Secrecy — Tại Sao Quan Trọng?

```
╔═══════════════════════════════════════════════════════════════╗
║   PERFECT FORWARD SECRECY (PFS)                              ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  KHÔNG CÓ PFS (RSA key exchange — đã bị loại khỏi TLS 1.3):║
║  → Server dùng CÙNG private key cho MỌI session            ║
║  → Hacker record traffic NGAY BÂY GIỜ (encrypted)          ║
║  → 5 năm sau, steal private key (hoặc quantum computer)     ║
║  → Decrypt TẤT CẢ traffic 5 năm trước!                     ║
║  → "Harvest now, decrypt later" attack!                      ║
║                                                               ║
║  CÓ PFS (ECDHE — TLS 1.3 BẮT BUỘC):                       ║
║  → Mỗi session tạo key pair MỚI (ephemeral!)               ║
║  → Session key XÓA ngay sau khi connection close!           ║

║  → Traffic ĐÃ RECORD → KHÔNG decrypt được! AN TOÀN!       ║
║                                                               ║
║  BÀI HỌC: LUÔN dùng ECDHE (không RSA key exchange!)       ║
║  → TLS 1.3 BẮT BUỘC ECDHE = always PFS!                   ║
```

### TLS 1.3 Handshake (1 RTT) — Nhanh Hơn 50%!

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   TLS 1.3 HANDSHAKE — CHỈ 1 RTT!                                        ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  Cải tiến so với TLS 1.2:                                                ║
║  → Client gửi key share NGAY trong ClientHello! (không đợi server!)    ║
║  → Server trả certificate + key share + Finished CÙNG LÚC!            ║
║  → 2 RTT → 1 RTT!                                                      ║
║                                                                           ║
║  Client                                           Server                  ║
║    │                                                 │                    ║
║    │  ═══════════ RTT 1 (DUY NHẤT!) ════════════   │                    ║
║    │                                                 │                    ║
║    │  ClientHello                                    │                    ║
║    │  ├── Supported versions: TLS 1.3               │                    ║
║    │  ├── Cipher suites: TLS_AES_256_GCM_SHA384     │                    ║
║    │  ├── Key share: ECDHE public value              │                    ║
║    │  │   (GỬI NGAY! Không đợi server!)             │                    ║
║    │  ├── SNI: api.example.com                       │                    ║
║    │  └── Supported groups: x25519, secp256r1        │                    ║
║    │────────────────────────────────────────────────►│                    ║
║    │                                                 │                    ║
║    │                               ServerHello       │                    ║
║    │                  ├── Key share (ECDHE)           │                    ║
║    │                  ├── Chosen cipher               │                    ║
║    │                  ┌─── ENCRYPTED from here! ───┐ │                    ║
║    │                  │ EncryptedExtensions         │ │                    ║
║    │                  │ Certificate                 │ │                    ║
║    │                  │ CertificateVerify           │ │                    ║
║    │                  │ Finished                    │ │                    ║
║    │                  └────────────────────────────┘ │                    ║
║    │◄────────────────────────────────────────────────│                    ║
║    │                                                 │                    ║
║    │  Finished (encrypted!)                           │                    ║
║    │────────────────────────────────────────────────►│                    ║
║    │                                                 │                    ║
║    │  ═══════════ DATA ═════════════                │                    ║
║    │  → Client CÓ THỂ gửi data NGAY sau ClientHello│                    ║
║    │    (early data / 0-RTT nếu đã connect trước!)  │                    ║
║    │                                                 │                    ║
║    │  Tổng: TCP (1 RTT) + TLS 1.3 (1 RTT) = 2 RTT  │                    ║
║    │  vs TLS 1.2: 3 RTT → tiết kiệm 33%!           │                    ║
║    │                                                 │                    ║
║  ╔════════════════════════════════════════════════════╗                   ║
║  ║  TLS 1.3 CẢI TIẾN CHÍNH:                         ║                   ║
║  ║  1. 1-RTT: client gửi key share ngay             ║                   ║
║  ║  2. 0-RTT: resumed connection gửi data ngay!     ║                   ║
║  ║  3. Encrypted handshake: cert cũng mã hóa!       ║                   ║
║  ║  4. Loại bỏ: RSA key exchange, CBC, SHA-1, RC4   ║                   ║
║  ║  5. Chỉ 5 cipher suites (vs 37+ trong TLS 1.2!) ║                   ║
║  ╚════════════════════════════════════════════════════╝                   ║
║                                                                           ║
╚═══════════════════════════════════════════════════════════════════════════╝
```

```go
// Go: mTLS client — service-to-service authentication
package main

import (
    "crypto/tls"
    "crypto/x509"
    "fmt"
    "io"
    "net/http"
    "os"
    "time"
)

func NewMTLSClient(certFile, keyFile, caFile string) (*http.Client, error) {
    // Load CLIENT certificate (mTLS: client cũng có cert!)
    clientCert, err := tls.LoadX509KeyPair(certFile, keyFile)
    if err != nil {
        return nil, fmt.Errorf("load client cert: %w", err)
    }

    // Load CA certificate (để verify SERVER)
    caCert, err := os.ReadFile(caFile)
    if err != nil {
        return nil, fmt.Errorf("read CA cert: %w", err)
    }
    caCertPool := x509.NewCertPool()
    caCertPool.AppendCertsFromPEM(caCert)

    return &http.Client{
        Timeout: 30 * time.Second,
        Transport: &http.Transport{
            TLSClientConfig: &tls.Config{
                Certificates: []tls.Certificate{clientCert}, // CLIENT cert!
                RootCAs:      caCertPool,                    // CA trust
                MinVersion:   tls.VersionTLS12,
            },
        },
    }, nil
}

func main() {
    client, err := NewMTLSClient(
        "client-cert.pem", "client-key.pem", "ca-cert.pem",
    )
    if err != nil {
        panic(err)
    }

    resp, err := client.Get("https://internal-service:8443/api/data")
    if err != nil {
        panic(err) // "tls: certificate required" nếu server yêu cầu mTLS!
    }
    defer resp.Body.Close()

    body, _ := io.ReadAll(resp.Body)
    fmt.Printf("Response: %s\n", body)
}
```

```go
// Go: CORS middleware production-ready
func corsMiddleware(allowedOrigins []string) func(http.Handler) http.Handler {
    originSet := make(map[string]bool)
    for _, o := range allowedOrigins {
        originSet[o] = true
    }

    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            origin := r.Header.Get("Origin")

            if originSet[origin] {
                w.Header().Set("Access-Control-Allow-Origin", origin)
                w.Header().Set("Access-Control-Allow-Credentials", "true")
                w.Header().Set("Vary", "Origin") // ⚠️ QUAN TRỌNG cho caching!
            }

            // Preflight
            if r.Method == "OPTIONS" {
                w.Header().Set("Access-Control-Allow-Methods",
                    "GET, POST, PUT, DELETE, OPTIONS")
                w.Header().Set("Access-Control-Allow-Headers",
                    "Authorization, Content-Type, X-Request-ID")
                w.Header().Set("Access-Control-Max-Age", "86400")
                w.WriteHeader(http.StatusNoContent)
                return
            }

            next.ServeHTTP(w, r)
        })
    }
}
```

### XSS (Cross-Site Scripting) — Tấn Công Phổ Biến Nhất

```
╔═══════════════════════════════════════════════════════════════╗
║   XSS — CROSS-SITE SCRIPTING                                 ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  CÁI GÌ: Hacker inject JavaScript vào website của BẠN!     ║
║  → JS chạy trong BROWSER của VICTIM!                         ║
║  → Đọc cookies, tokens, localStorage → GỬI VỀ HACKER!     ║
║                                                               ║
║  3 LOẠI XSS:                                                 ║
║  ──────────                                                  ║
║                                                               ║
║  1. Stored XSS (Persistent):                                ║
║     → Hacker gửi comment: <script>steal(cookies)</script>   ║
║     → Server LƯU VÀO DATABASE!                              ║
║     → Mọi user mở page → script chạy → cookies bị steal!  ║
║     → NGUY HIỂM NHẤT! Ảnh hưởng TẤT CẢ users!            ║
║                                                               ║
║  2. Reflected XSS:                                           ║
║     → Hacker gửi link: example.com/search?q=<script>...    ║
║     → Server trả: "Kết quả cho: <script>..."               ║
║     → Script chạy trong browser victim!                      ║
║     → Cần victim click link (social engineering)             ║
║                                                               ║
║  3. DOM-based XSS:                                           ║
║     → JavaScript client-side xử lý input không an toàn     ║
║     → document.innerHTML = userInput → XSS!                 ║
║     → Server không liên quan, lỗi ở CLIENT code!           ║
║                                                               ║
║  PHÒNG CHỐNG:                                                 ║
║  ────────────                                                ║
║  ✅ Output encoding: HTML-encode TẤT CẢ user input         ║
║     → < → &lt;  > → &gt;  " → &quot;                       ║
║     → Go: html/template TỰ ĐỘNG escape!                    ║
║  ✅ Content-Security-Policy header:                          ║
║     → CSP: script-src 'self' → chỉ chạy script từ domain! ║
║     → Chặn inline scripts (<script>evil</script>)           ║
║  ✅ HttpOnly cookies: JS KHÔNG đọc được cookies!            ║
║  ✅ Input validation: whitelist, không blacklist!            ║
║  ❌ ĐỪNG: innerHTML = userInput (DOM XSS!)                  ║
║  ❌ ĐỪNG: trust user input EVER!                            ║
```

### CSRF (Cross-Site Request Forgery)

```
╔═══════════════════════════════════════════════════════════════╗
║   CSRF — CROSS-SITE REQUEST FORGERY                          ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  CÁI GÌ: Hacker lừa browser CỦA BẠN gửi request đến     ║
║  website mà bạn ĐÃ LOGIN, sử dụng COOKIES CỦA BẠN!       ║
║                                                               ║
║  CÁCH HOẠT ĐỘNG:                                             ║
║  ────────────────                                            ║
║  1. Victim đã login bank.com (cookie session=abc123)        ║
║  2. Victim visit evil.com (hacker's site)                    ║
║  3. evil.com có hidden form:                                 ║
║     <form action="https://bank.com/transfer" method="POST"> ║
║       <input name="to" value="hacker-account">              ║
║       <input name="amount" value="10000">                   ║
║     </form>                                                   ║
║     <script>document.forms[0].submit();</script>            ║
║  4. Browser GỬI request đến bank.com VỚI COOKIE CỦA VICTIM!║
║  5. Bank.com thấy valid cookie → CHUYỂN TIỀN!              ║
║                                                               ║
║  TẠI SAO HOẠT ĐỘNG?                                         ║
║  → Browser TỰ ĐỘNG gửi cookies cho domain đích!            ║

║                                                               ║
║  PHÒNG CHỐNG:                                                 ║
║  ────────────                                                ║
║  ✅ SameSite cookies (Lax/Strict):                           ║
║     → SameSite=Lax: cookie KHÔNG gửi khi POST cross-site!  ║
║     → Chrome DEFAULT = Lax → CSRF khó hơn nhiều!          ║
║  ✅ CSRF token:                                               ║
║     → Server generate random token → embed trong form       ║
║     → Server verify token mỗi POST request                  ║
║     → Hacker KHÔNG BIẾT token → không forge được!          ║
║  ✅ Check Origin/Referer header:                             ║
║     → Nếu Origin ≠ expected → REJECT!                      ║
║  ✅ Custom request headers (X-CSRF-Token):                   ║
║     → fetch() có thể add custom headers                     ║
║     → Form submission KHÔNG THỂ → CSRF thất bại!          ║
║                                                               ║
```

### Kịch Bản Thất Bại: "Khi Nào Security Sẽ Fail?"

```
╔═══════════════════════════════════════════════════════════════╗
║   SECURITY FAILURE SCENARIOS                                  ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  1. CERTIFICATE EXPIRY (90 ngày Let's Encrypt!):            ║
║     → Quên renew → site DOWN với "Not Secure"!              ║
║     → Fix: auto-renewal (certbot, acme.sh)                  ║
║     → Monitor: cert.expiryDate < 14 days → ALERT!           ║
║                                                               ║
║  2. PRIVATE KEY COMPROMISE:                                   ║
║     → Key trong git repo, env vars leaked, dev machine hack  ║
║     → Fix: HSM/Vault, rotate keys, revoke old certs!        ║
║     → Forward secrecy (ECDHE): past traffic = still safe!   ║
║                                                               ║
║  3. TLS DOWNGRADE ATTACK:                                     ║
║     → MITM forces TLS 1.0 → exploit known vulnerabilities  ║
║     → Fix: MinVersion = TLS 1.2, HSTS preload              ║
║     → TLS 1.3: anti-downgrade mechanism built-in!           ║
║                                                               ║
║  4. SUBDOMAIN TAKEOVER:                                       ║
║     → CNAME đến service đã xóa (old Heroku, S3 bucket)     ║
║     → Hacker claim subdomain → valid cert → phishing!      ║

║                                                               ║
║  5. TOKEN/SESSION LEAK:                                       ║
║     → JWT in URL query params → log files, referer header!  ║
║     → Session cookie without HttpOnly → XSS steal!         ║
║     → Fix: tokens in headers ONLY, HttpOnly cookies!        ║
║                                                               ║
║  6. SUPPLY CHAIN ATTACK:                                      ║
║     → npm package bị hack → malicious code trong build!    ║
║     → Fix: lock dependencies, audit, SRI for CDN scripts!  ║
║                                                               ║
```

## §7. Debug TLS — Practical Commands

```bash
# ═══════════════════════════════════════════════════
# TLS CERTIFICATE INSPECTION
# ═══════════════════════════════════════════════════

openssl s_client -connect api.example.com:443 -servername api.example.com < /dev/null 2>/dev/null | openssl x509 -noout -text | head -20
```
