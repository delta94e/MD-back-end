## Contextual History — Tại Sao DNS Tồn Tại?

### Câu chuyện: Từ hosts.txt đến DNS toàn cầu

**Những năm 1970 — Internet chỉ có vài trăm máy:**

Mỗi máy tính trên ARPANET có 1 file duy nhất: `/etc/hosts` (hoặc `HOSTS.TXT`). File này chứa TOÀN BỘ mapping tên→IP của MỌI máy trên mạng. Mỗi khi có máy mới, Stanford Research Institute (SRI-NIC) cập nhật file và MỌI NGƯỜI phải download lại!

```
1970s ─── HOSTS.TXT — 1 file cho TOÀN BỘ internet!
  │       → SRI-NIC duy trì, mọi người FTP download
  │       → Vài trăm máy → OK, quản lý được!
  │
1983 ─── Internet BÙNG NỔ! Hàng nghìn máy → HOSTS.TXT THẤT BẠI!
  │       → File quá lớn, download mất thời gian
  │       → Name conflicts: ai cũng muốn đặt "mail" hay "server"!
  │       → Single point of failure: SRI-NIC down = không resolve!
  │       → Consistency: file cũ ngay khi download xong!
  │
1983 ─── Paul Mockapetris phát minh DNS (RFC 882, 883)
  │       → PHÂN TÁN: không 1 server nào giữ TẤT CẢ!
  │       → PHÂN CẤP: Root → TLD → Domain → Subdomain
  │       → CACHING: không cần hỏi lại mỗi lần!
  │       → DELEGATION: mỗi tổ chức quản lý zone của mình!
  │
1987 ─── RFC 1034/1035 — DNS specification chuẩn (VẪN DÙNG!)
  │       → Định nghĩa: record types, zone file format, query protocol
  │       → UDP port 53 (queries), TCP port 53 (zone transfer, large)
  │
1997 ─── BIND (Berkeley Internet Name Domain) trở thành dominant
  │       → DNS software phổ biến nhất, vẫn dùng đến hiện tại
  │
2010 ─── DNSSEC deployment bắt đầu phổ biến
  │       → Ký DNS records để chống spoofing
  │       → Nhưng: phức tạp, performance overhead!
  │
2018 ─── DNS over HTTPS (DoH) — RFC 8484
  │       → Encrypt DNS queries! ISP không biết bạn query gì!
  │       → Cloudflare 1.1.1.1, Google 8.8.8.8 hỗ trợ
  │
2025 ─── Hiện tại:
         → ~350 TRIỆU domain names registered
         → ~1.1 NGHÌN TỶ DNS queries/ngày
         → 13 root server clusters (nhưng > 1500 instances!)
         → DNS vẫn dùng spec TỪ 1987! (backward compatible!)
```

```
║  → Mỗi level chỉ biết level TIẾP THEO!                     ║
║  → Root: chỉ biết TLD servers                                ║

║  → Auth: chỉ biết records CỦA NÓ!                          ║
║                                                               ║
```

## 5 Whys: DNS Resolution

```
╔═══════════════════════════════════════════════════════════════╗
║   5 WHYS — DNS RESOLUTION                                    ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  WHY 1: "Tại sao cần DNS?"                                  ║
║  → Humans nhớ TÊN (google.com), computers dùng SỐ (IP)!    ║
║  → 142.250.191.46 vs google.com — cái nào dễ nhớ?          ║
║  → IP có thể THAY ĐỔI (server migration, load balancing)   ║
║  → Domain name = STABLE identifier, IP = implementation!    ║
║                                                               ║
║  WHY 2: "Tại sao DNS query qua UDP mà không TCP?"          ║
║  → DNS query/response thường < 512 bytes (1 UDP packet!)    ║
║  → TCP cần 3-way handshake = 1.5 RTT TRƯỚC KHI query!     ║
║  → UDP: gửi query ngay = 0.5 RTT!                          ║
║  → Tiết kiệm 67% latency cho MỖI DNS query!               ║
║  → ⚠️ Response > 512 bytes → fallback TCP (EDNS0: 4096)   ║
║  → ⚠️ Zone transfer (AXFR) LUÔN dùng TCP → reliability!  ║
║                                                               ║
║  WHY 3: "Tại sao cần TTL? Sao không cache mãi?"            ║
║  → Records THAY ĐỔI! Server migrate, IP đổi, failover!    ║
║  → Cache mãi = trỏ đến IP CŨ = DOWN!                      ║
║  → TTL = "sau N giây, hỏi LẠI!"                           ║
║  → TTL ngắn (60s): update nhanh, NHƯNG nhiều queries hơn!  ║
║  → TTL dài (86400s): ít queries, NHƯNG update chậm!        ║
║  → Trade-off: freshness vs performance!                      ║
║                                                               ║
║  WHY 4: "Tại sao 13 root servers MÀ internet không sập?"   ║
║  → 13 là SỐ ROOT SERVER CLUSTERS (A-M), không phải 13 máy!║
║  → Thực tế: > 1,500 instances TOÀN CẦU (anycast!)         ║
║  → Anycast: cùng 1 IP, NHIỀU servers, routing đến GẦN NHẤT║
║  → Root server J (Verisign): 200+ locations trên thế giới! ║
║  → Tại sao 13? Giới hạn kích thước UDP packet (512 bytes) ║
║    → 13 NS records + glue records vừa đủ 1 packet!        ║
║                                                               ║
║  WHY 5: "Tại sao DNS propagation mất thời gian?"           ║
║  → KHÔNG PHẢI propagation! Mà là CACHE EXPIRY!             ║
║  → Bạn đổi IP → old TTL chưa hết → resolvers trả IP CŨ! ║
║  → TTL = 3600 (1h) → worst case: 1h mới update!           ║
║  → ISP resolver: đôi khi IGNORE TTL (cache lâu hơn!)      ║
║  → Fix: hạ TTL TRƯỚC khi migration (300s → 60s → deploy!) ║
║                                                               ║
╚═══════════════════════════════════════════════════════════════╝
```

## §2. DNS Record Types — Chi Tiết

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   DNS RECORD TYPES — BACKEND ENGINEER CẦN BIẾT                          ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  A RECORD (Address):                                                      ║
║  → Domain → IPv4 address                                                ║
║  → api.example.com → 93.184.216.34                                     ║
║  → Có thể có NHIỀU A records (DNS round-robin!)                        ║
║  → api.example.com → 93.184.216.34                                     ║
║                       93.184.216.35                                      ║
║                       93.184.216.36                                      ║
║  → Resolver trả TẤT CẢ, client chọn 1 (hoặc rotate!)                ║
║                                                                           ║
║  AAAA RECORD (IPv6 Address):                                             ║
║  → Domain → IPv6 address                                                ║
║  → api.example.com → 2606:2800:220:1:248:1893:25c8:1946               ║
║  → Giống A record nhưng cho IPv6!                                       ║
║                                                                           ║
║  CNAME RECORD (Canonical Name — Alias):                                  ║
║  → Domain → ANOTHER domain (alias!)                                     ║
║  → www.example.com → example.com (CNAME)                               ║
║  → cdn.example.com → d1234.cloudfront.net (CNAME → CDN!)              ║
║  → ⚠️ CNAME KHÔNG THỂ ở root domain (example.com)!                   ║
║  → ⚠️ CNAME giải quyết → thêm 1 DNS lookup → chậm hơn!             ║
║  → Giải pháp root: ALIAS/ANAME record (Cloudflare, Route53)           ║
║                                                                           ║
║  MX RECORD (Mail Exchange):                                              ║
║  → Domain → mail server (with priority!)                                ║
║  → example.com MX 10 mail1.example.com                                  ║
║  →             MX 20 mail2.example.com                                  ║
║  → Priority nhỏ = ưu tiên cao! (10 trước 20)                          ║
║  → Fallback: mail1 down → gửi đến mail2!                              ║
║                                                                           ║
║  NS RECORD (Nameserver):                                                  ║
║  → Domain → authoritative nameserver                                    ║
║  → example.com NS ns1.example.com                                       ║
║  →             NS ns2.example.com                                       ║
║  → PHẢI có ≥2 NS (redundancy!)                                        ║
║                                                                           ║
║  TXT RECORD (Text — đa năng!):                                          ║
║  → Chứa text string bất kỳ                                             ║
║  → SPF: "v=spf1 include:_spf.google.com ~all" (email auth!)           ║
║  → DKIM: email signature verification                                    ║
║  → DMARC: "v=DMARC1; p=reject" (email policy!)                        ║
║  → Domain verification: Google, Let's Encrypt, etc.                     ║
║  → ⚠️ Max 255 chars per string, nhưng multiple strings OK             ║
║                                                                           ║
║  SRV RECORD (Service — service discovery!):                              ║
║  → _service._protocol.domain → host:port (+ priority + weight)        ║
║  → _http._tcp.example.com SRV 10 60 8080 api1.example.com              ║
║  → Kubernetes sử dụng SRV cho service discovery!                       ║
║                                                                           ║
║  CAA RECORD (Certificate Authority Authorization):                       ║
║  → Chỉ định CA NÀO được phát hành certificate cho domain!             ║
║  → example.com CAA 0 issue "letsencrypt.org"                            ║
║  → ⚠️ Nếu không có CAA → BẤT KỲ CA nào đều issue được!             ║
║  → → BẮT BUỘC nên có để bảo vệ domain!                               ║
║                                                                           ║
║  PTR RECORD (Pointer — reverse DNS):                                     ║
║  → IP → domain (ngược lại A record!)                                   ║
║  → 34.216.184.93.in-addr.arpa → api.example.com                        ║
║  → Dùng cho: email server verification, logging, security              ║
║                                                                           ║
╚═══════════════════════════════════════════════════════════════════════════╝
```

## §3. DNS Caching & TTL — Trade-off Analysis

```
╔═══════════════════════════════════════════════════════════════╗
║   TTL STRATEGY — TRADE-OFF MATRIX                            ║
╠═══════════════════════════════════════════════════════════════╣
║                                                               ║
║  ┌──────────┬─────────────────┬──────────────────────┐       ║
║  │ TTL      │ Ưu điểm        │ Nhược điểm           │       ║
║  ├──────────┼─────────────────┼──────────────────────┤       ║
║  │ 60s      │ Update nhanh!   │ Nhiều queries → cost!│       ║
║  │ (1 min)  │ Failover nhanh! │ Latency tăng!        │       ║
║  │          │ Migration dễ!   │ Root/TLD load tăng!  │       ║
║  ├──────────┼─────────────────┼──────────────────────┤       ║
║  │ 300s     │ Balance tốt!    │ 5 phút delay update │       ║
║  │ (5 min)  │ Đủ nhanh cho   │                      │       ║
║  │          │ hầu hết cases! │                      │       ║
║  ├──────────┼─────────────────┼──────────────────────┤       ║
║  │ 3600s    │ Ít queries!     │ 1 giờ delay update! │       ║
║  │ (1 hour) │ Cache hiệu quả │ Migration chậm!       │       ║
║  ├──────────┼─────────────────┼──────────────────────┤       ║
║  │ 86400s   │ Rất ít queries! │ 1 NGÀY delay update!│       ║
║  │ (1 day)  │ Best performance│ Không thể failover! │       ║
║  └──────────┴─────────────────┴──────────────────────┘       ║
║                                                               ║
║  BEST PRACTICES:                                              ║
║  → Primary API domain: TTL 300s (5 min)                      ║
║  → CDN/static assets:  TTL 3600-86400s                       ║
║  → Trước migration:    Hạ TTL → 60s (24h TRƯỚC!)           ║
║  → Sau migration xong: Tăng TTL lại 300-3600s              ║
║  → Failover domain:    TTL 60s (luôn thấp!)                ║
║                                                               ║
║  MIGRATION WORKFLOW:                                          ║
║  ──────────────────                                          ║
║  T-24h: Hạ TTL từ 3600 → 60 (đợi old TTL hết!)           ║
║  T-0:   Đổi A record sang IP mới                           ║
║  T+5m:  Kiểm tra: dig +trace api.example.com               ║
║  T+1h:  Xác nhận mọi traffic đến IP mới                    ║
║  T+24h: Tăng TTL lại 300-3600                              ║
║                                                               ║
╚═══════════════════════════════════════════════════════════════╝
```

## §4. DNS Load Balancing & High Availability

```
╔═══════════════════════════════════════════════════════════════════════════╗
║   DNS-BASED LOAD BALANCING                                               ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  1. ROUND-ROBIN (đơn giản nhất):                                       ║
║  ──────────────────────────────                                          ║
║  api.example.com  A  10.0.1.1                                            ║
║                   A  10.0.1.2                                            ║
║                   A  10.0.1.3                                            ║
║                                                                           ║
║  Query 1: trả [10.0.1.1, 10.0.1.2, 10.0.1.3]                          ║
║  Query 2: trả [10.0.1.2, 10.0.1.3, 10.0.1.1] (rotated!)              ║
║  → Client thường chọn IP ĐẦU TIÊN → phân tải!                        ║
║  → ⚠️ Không health check! Server down vẫn trả IP!                    ║
║  → ⚠️ Caching: resolver cache = cùng IP trong TTL!                   ║
║                                                                           ║
```

## §6. Reverse Engineering — DNS Tools & Go Implementation

```go
// Go: DNS resolver from scratch (educational!)
// "What I cannot create, I do not understand" — Feynman
package main

import (
    "encoding/binary"
    "fmt"
    "math/rand"
    "net"
    "strings"
)

// DNS Header (12 bytes)
type DNSHeader struct {
    ID      uint16 // Random query ID
    Flags   uint16 // QR, Opcode, AA, TC, RD, RA, RCODE
    QDCount uint16 // Number of questions
    ANCount uint16 // Number of answers
    NSCount uint16 // Number of authority records
    ARCount uint16 // Number of additional records
}

// Encode domain name to DNS wire format
// "api.example.com" → [3]api[7]example[3]com[0]
func encodeDomainName(domain string) []byte {
    var result []byte
    parts := strings.Split(domain, ".")
    for _, part := range parts {
        result = append(result, byte(len(part)))
        result = append(result, []byte(part)...)
    }
    result = append(result, 0) // null terminator
    return result
}

func buildDNSQuery(domain string) []byte {
    // Header
    header := DNSHeader{
        ID:      uint16(rand.Intn(65535)),
        Flags:   0x0100, // RD (Recursion Desired) = 1
        QDCount: 1,      // 1 question
    }

    buf := make([]byte, 12)
    binary.BigEndian.PutUint16(buf[0:], header.ID)
    binary.BigEndian.PutUint16(buf[2:], header.Flags)
    binary.BigEndian.PutUint16(buf[4:], header.QDCount)
    // ANCount, NSCount, ARCount = 0

    // Question section
    buf = append(buf, encodeDomainName(domain)...)
    buf = append(buf, 0, 1) // QTYPE: A (1)
    buf = append(buf, 0, 1) // QCLASS: IN (1)

    return buf
}

func main() {
    domain := "example.com"
    query := buildDNSQuery(domain)

    // Send to Google Public DNS (8.8.8.8:53)
    conn, err := net.Dial("udp", "8.8.8.8:53")
    if err != nil {
        panic(err)
    }
    defer conn.Close()

    conn.Write(query)

    response := make([]byte, 512)
    n, err := conn.Read(response)
    if err != nil {
        panic(err)
    }

    // Parse response (simplified)
    anCount := binary.BigEndian.Uint16(response[6:8])
    fmt.Printf("Domain: %s\n", domain)
    fmt.Printf("Response size: %d bytes\n", n)
    fmt.Printf("Answer count: %d\n", anCount)

    // Skip header (12) + question section → find answers
    // In production, use net.LookupHost() or miekg/dns library!
    if anCount > 0 {
        // Simple: parse the last 4 bytes as IPv4 for A record
        ip := net.IPv4(response[n-4], response[n-3], response[n-2], response[n-1])
        fmt.Printf("IP: %s\n", ip)
    }
}

// BÀI HỌC:
// 1. DNS query = UDP packet, rất nhỏ (< 100 bytes cho simple query)
// 2. Domain names encoded: length-prefixed labels, null terminated
// 3. Header chỉ 12 bytes: ID, flags, counts
// 4. BigEndian byte order (network byte order!)
// 5. Production: dùng net.LookupHost() hoặc miekg/dns library!
```
